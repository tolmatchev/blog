## Valotile

Оператор volatile гарантирует следуещее 
-  Операции *присваивания и получений* для этого поля атомарны и видны всем остальным потокам, запрещаются определнные оптимизации, связанные с кешированием этих переменных.
-  *Порядок выполнения* операций с этой переменной гарантируется

> Но при этом он не гарантирует того, что составные операции будут атомарным над этим полем, например:
```java
x++;
x = x + 1;
if (x == 0) {
x = 1 }
```

Т.е. volatile переменные подходят для следущих случаев, когда переменную изменяет один поток, а увидеть должны све остальные

Также стоит добавить, что до Java 5 операция чтения и записи была **неатомарная** для полей типа `long` и `double` из-за того что Java изначально писалась для *32-битной архитектуры* и эти операции выполнялись в 2 атомарные операции и другой поток мог увидеть половину значения из старой переменной, а другую половину из новой.

Поэтому, начиная с 5 Java была полностью переписана `Java Memory Model` и введена модель отношения `Happens-before`.

## Happens-before

Это центральное отношение модели памяти `Java Memory Model`, которое гарантирует, *порядок выполнения* операций и *видимость изменений*, для других потоков. Другими словами, если *A выполняется перед B* то все то все действия в A будут *гарантированно видны* в B и A будет происходить логически раньше B 

### Достигнуть `happens-before` можно разными способами:

Оператор `volatile`, `synchronized`, `Lock`, `Atomic`, `Thread Api`. Один из самых простых способов, что обеспечит видимость изменений для других потоков, это объявить как раз переменную с типом `volatile`
