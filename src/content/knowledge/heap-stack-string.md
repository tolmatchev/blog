### Что такое _Heap_ и _Stack_ память в Java? Какая разница между ними? 

Куча - это область памяти JVM, где хранятся все объекты Java и связанные с ними данные. Выделение памяти в куче происходит динамически во время выполнения программы.

Каждый поток в Java имеет собственный стек, который содержит информацию о вызовах методов и локальных переменных. Стек устроен по [принципу LIFO](https://ru.wikipedia.org/wiki/LIFO) (last-in, first-out) и используется для управления выполнением методов и их локальными переменными.

- **Heap (куча)** используется Java Runtime для выделения памяти под объекты и классы. Создание нового объекта также происходит в куче. Это же является областью работы сборщика мусора. Любой объект, созданный в куче, имеет глобальный доступ и на него могут ссылаться из любой части приложения.


### Различия между _Heap_ и _Stack_ памятью:

- Куча используется всеми частями приложения, в то время как стек используется только одним потоком исполнения программы. ( [[Concurrency from video]] *пункт с синхронизацией* )
- Всякий раз, когда создается объект, он всегда хранится в куче, а в памяти стека содержится лишь ссылка на него. Память стека содержит только локальные переменные примитивных типов и ссылки на объекты в куче.
- Объекты в куче доступны с любой точки программы, в то время как стековая память не может быть доступна для других потоков. ( [[Concurrency from video]] *пункт с синхронизацией* )
- Стековая память существует лишь какое-то время работы программы, а память в куче живет с самого начала до конца работы программы.
- Если память стека полностью занята, то Java Runtime бросает исключение `java.lang.StackOverflowError`. Если заполнена память кучи, то бросается исключение `java.lang.OutOfMemoryError: Java Heap Space`.
- Размер памяти стека намного меньше памяти в куче.
- Из-за простоты распределения памяти, стековая память работает намного быстрее кучи.

Для определения начального и максимального размера памяти в куче используются `-Xms` и `-Xmx` опции JVM. Для стека определить размер памяти можно с помощью опции `-Xss`.

### Иммутабельность `String`

Вот какие преимущества несет собой имутабельность `String`
-  **Безопасность**. 
   Если строки изменяемые, то может произойти непреднамернное ее изменение 
  ```java
  String password = "qwerty";
  password.CharAt(0) = r;
  System.out.println(password) /// rwerty
  ```

  > Также могут теряться ключи в `HashMap` (стандартная задачка на собесах)
  
- **Потокобезопасность**
  Из-за того, что их нельзя менять, не будет `race condition`
- **Кэширование и интернирование**. В Heap есть специальная область памяти, где хранятся уникальные строки. По умолчанию там хранятся только литералы. С помощью функции `intern()` можно свзяать строку с `String Pool` и это будет гарантировать, что теперь там будет только один такой объект. Делать нужно это крайне аккуратно, ибо неправильное интернирование введет к к разрастранию `Old Gen` (вопрос в почему?). Получаем что строки можем сравнивать ***по ссылке***

  ```java
  	String heap = new String("Hello"); // объект в heap
  	String pool = "Hello";             // объект в String Pool
  	
  	System.out.println(heap == pool); // false
  	
  	String interned = heap.intern();  
  	System.out.println(interned == pool); // true (оба из пула)
  ``` 


**Строки — идеальные кандидаты для ключей в `HashMap`**, но не потому что JVM всегда сравнивает их по ссылке, а потому что:

1. Они иммутабельны,
2. У них кэшированный `hashCode`
3. `equals()` оптимизирован
4. Интернирование дополнительно помогает экономить память и ускоряет сравнение (через быстрый `==` внутри `equals`).

| Код                                        | Где         | Комментарий                                                             |
| ------------------------------------------ | ----------- | ----------------------------------------------------------------------- |
| String s = "abc"                           | String Pool | Такие строковые <br>литералы на этапе<br>компиляции <br>помещаются туда |
| String s = new String("abc")               | Heap        | принудитиельно в heap                                                   |
| String s = "a" + "b"                       | String Pool | Фактически превращается в <br>String s = "ab"                           |
| String a = "str";<br>String b = a + "ing"; | Heap        | Конкатенация выполняется во время выполнения программы                  |
| .intern()                                  | String pool | Интернирование строк принудительное                                     |
