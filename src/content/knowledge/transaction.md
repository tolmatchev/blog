*Транзакция* - это набор неделимых операций, объединенных в одну атомарную, 
что значит она либо выполняется полностью, либо вообще не выполняется   

## Уровни изоляции транзакций

`READ UNCOMITTED` - можем читать даже те записи, которые не закомичены, вследствие этого, получаем не акутальные данные и допускаем грязное чтение *dirty read*

`READ COMMITTED` - Читаем только закомиченное. Стоит в большинстве реляционных СУБД по умлочанию. Пример того, какие проблемы бывают с таким уровням изоляции, например 

### Фантомное чтение

| Транзакция 1                       | Транзакция 2                     |
| ---------------------------------- | -------------------------------- |
| `select count(*) from emails -> 2` |                                  |
|                                    | `insert values(...) into emails` |
| `select * from emails -> 3`        |                                  |

### non-repeatable-read

| Транзакция 1                                      | Транзакция 2                                    |
| ------------------------------------------------- | ----------------------------------------------- |
| `select balance from account where id = 1 -> 100` |                                                 |
|                                                   | `update account set balance = 200 where id = 1` |
| `select balance from account where id = 1 -> 200` |                                                 |


`REPEATABLE READ (SNAPSHOT)` - во время старта транзакции создается, так называемый *слепок базы данных*, с которым работает только данная транзакция. Параллельные транзакции  при этом могут выполнять, но этих изменения не увидит наша транзакция. Поэтому решает проблему *non-repeatable-read* (всегда) и *фантомного чтения* (в большинстве случаев, но все таки зависит от конкретной СУБД)

| Транзакция 1                       | Транзакция 2                     |
| ---------------------------------- | -------------------------------- |
| `select count(*) from emails -> 2` |                                  |
|                                    | `insert values(...) into emails` |
| `select * from emails -> 2`        |                                  |
`SERIALIZIBLE` - Самая строгий и дорогой уровень изоляций. Если есть параллельная транзакция, то этот уровень изоляции эквивалентен их последовательному выполнению, он учитывает **взаимосвязи между запросами** и гарантирует соблюдение **глобальных инвариантов**, даже если каждая транзакция по отдельности корректна. Этот уровень изоляции отлавливает аномалии и разными способами: блокировка параллельных транзакций, их откат, возвращать ошибку сериализации 

| Транзакция 1                                           | Транзакция 2                                           |
| ------------------------------------------------------ | ------------------------------------------------------ |
| `select count(*) from users where role = 'admin' -> 2` |                                                        |
|                                                        | `select count(*) from users where role = 'admin' -> 2` |
| `update users set role = 'user' where id = 1`          |                                                        |
|                                                        | `update users set role = 'user' where id = 2`          |
| `commit`                                               |                                                        |
|                                                        | ❌ ошибка сериализации / откат                          |
