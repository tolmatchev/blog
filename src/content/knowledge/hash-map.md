### 1. **Вызывается метод `put(K key, V value)`**

Когда мы добавляем новый элемент, вызывается метод `put(K key, V value)`, где `K` — ключ, а `V` — значение, которое нужно сохранить.

### 2. **Вычисление хэш-кода ключа**

Сначала вызывается метод `hashCode()` на ключе. 
Он возвращает целочисленное значение, которое затем используется для определения индекса ведра (bucket), где будет храниться элемент.

`
Integer hash = hash(key.hashCode());
`

Метод `hash()` в `HashMap` модифицирует полученный хэш-код для уменьшения коллизий, изменяя его биты, чтобы обеспечить лучшее распределение по ведрам, особенно для случаев с неравномерно распределенными хэшами. 
> 
> (Вот тут важный момент с тем, а какие например такие ключи могут быть - например `Integer`)
> 

### 3. **Определение индекса ведра**

Индекс ведра вычисляется по формуле:

`
Integer bucketIndex = (n - 1) & hash;
`

Здесь `n` — текущая ёмкость массива ведер в `HashMap`, а `&` — побитовое AND, которое эффективно превращает хэш в допустимый индекс массива. Это позволяет распределять элементы по всему массиву, а не скапливать их в одном месте.

### 4. **Проверка ведра на наличие элементов**

Если по рассчитанному индексу ведро пустое, создается новый узел (Node) и добавляется в это ведро. Если ведро уже содержит один или несколько элементов (коллизия), то выполняется одна из следующих операций:

- **Если это ==односвязный список==**: происходит перебор элементов списка, чтобы проверить, существует ли уже элемент с таким же ключом. Если такой ключ найден, значение обновляется. Если нет, то новый элемент добавляется в конец списка.
- **Если это ==бинарное дерево==** (с Java 8 и выше, при превышении порога в 8 элементов в одном ведре): ведро преобразуется в бинарное дерево для оптимизации времени поиска, и элемент вставляется в дерево.

### 5. **Решение проблемы коллизий (связывание или дерево)**

При коллизии элементы сначала добавляются в виде связного списка. В Java 8 и выше, если количество элементов в одном ведре превышает пороговое значение (по умолчанию — 8), то связанный список преобразуется в сбалансированное бинарное дерево, что улучшает сложность поиска с O(n) до O(log⁡n).

### 6. **Проверка на превышение порога загрузки (load factor)**

После добавления элемента `HashMap` проверяет, не превышена ли максимальная загруженность таблицы. Порог загрузки (==по умолчанию 0.75 и кол-во ведер 16==)  определяет, на сколько процентов может быть заполнена таблица до того, как она будет увеличена. Если количество элементов превышает пороговое значение, таблица увеличивается в два раза (ресайзинг).

### 7. **Ресайзинг и перераспределение элементов**

При увеличении таблицы в два раза все элементы пересчитываются и распределяются по новой таблице, так как изменяется количество ведер. Это нужно для поддержания равномерного распределения и минимизации коллизий.
